# PR4571: Preprocess Gas Accounting and Benchmarking

## Context

GnoVM's `Preprocess` pass does substantial work before execution: type
checking, name resolution, constant folding, block initialization, recursive
predefinition, and transcription. The cost of this work scales with the shape
of the code being preprocessed â€” a contract full of `*StructTypeExpr`
declarations is materially more expensive to preprocess than one full of
`*NameExpr`s â€” but none of it was being billed to the transaction's gas
meter. That created two problems:

1. **Under-charge / DoS surface.** An adversary could submit a transaction
   that spent most of its work in the preprocess pass (complex types, deeply
   nested composites) and pay little to no gas for it.
2. **No measurement surface.** There was no way to attribute wall-clock time
   to specific preprocess operations, so we could neither tune costs nor
   detect regressions.

The `benchops` package already measures three categories â€” VM opcode
execution (`CPUOp`), store operations (`StoreOp`), and native function calls
(`NativeOp`) â€” using a single shared timeline (`curOpCode byte` +
`finalizeCurrent()`). This PR extends that framework with a fourth category
â€” **preprocess operations** (`PreprocessOp`) â€” that runs on its own
independent timer pair (`preprocessCounts` / `preprocessAccumDur`) with a
frame stack for recursive preprocessing.

This PR is a reauthor of the stale branch `feat/add_gas_preprocess` which
had diverged from master across three major upstream refactors:
- PR #5291 (Xeon 8168 gas calibration) â€” rewrote the benchmarking tooling
  from a standalone `cmd/benchops` binary (deleted) to `go test -bench`
  with a Python analysis pipeline in `cmd/calibrate/`.
- PR #5415 (gas storage refactor) â€” changed `Store.BeginTransaction`
  signature, touched many files.
- PR #5544 (type persistence dedup) â€” breaking change in type storage.

Rather than resolve the 85 merge conflicts mechanically, the branch was
rebuilt from scratch against current master, adopting the new bench
pattern (Go benchmarks in `bench_*_test.go` files) instead of the
now-removed custom binary.

## Decision

### Part A: Extend `benchops` with a fourth category

Add `PreprocessOp` (one code per reachable (`TransStage`, `Node` type) pair
that Transcribe dispatches) in a new `pkg/benchops/preprocess.go` file.
That file contains:

- `PreprocessOp` type, code constants, names table, `PreprocessCodeString`.
- Timing state (`preprocessCounts`, `preprocessAccumDur`,
  `preprocessStack`) kept separate from the main op timeline, so the
  preprocess timer doesn't interact with VM op timing.
- `StartPreprocess(code)` / `StopPreprocess()` with a frame stack â€” pushes
  the outer measurement when a nested `Preprocess` is invoked and resumes
  it on return. This handles recursive preprocessing (e.g., sub-tree
  re-preprocessing after constant folding) correctly.
- Read-back helpers (`PreprocessAccumDur`, `PreprocessCount`,
  `ResetPreprocess`) used by the calibration benchmark.

The instrumentation is gated by a package-level `preprocessTiming` bool
in `gnolang`, exposed via `SetPreprocessTiming`. In production this stays
`false` and the Start/Stop calls are a single branch-predicted check; the
calibration benchmark flips it on for the duration of its run.

### Part B: Thread `gasMeter` through preprocess

`Preprocess` gains a `gasMeter store.GasMeter` parameter. It's threaded
through every helper that performs preprocess-stage work: `preprocess1`,
`checkOrConvertType`, `convertType`, `doConvertType`, `findUndefinedV/T/Any`,
`predefineRecursively`, `predefineRecursively2`, `tryPredefine`,
`defineOrDecl`, `parseAssignFromExprList`, `checkOrConvertIntegerKind`.
External callers pass `m.GasMeter` from machine sites or `nil` from
non-transaction contexts (tests, REPL, stdlib predefine).

### Part C: Gas cost table, calibrated from Go-bench

`preprocessGasCosts [256]store.Gas` maps each `PreprocessOp` to its gas cost
in nanoseconds. The values are regenerated by running:

```
make -C gnovm calibrate.preprocess              # on Xeon 8168 (reference)
make -C gnovm calibrate.preprocess SCALE=1.5    # on Apple M1 Pro
```

The benchmark is `BenchmarkPreprocess_Corpus` in
`pkg/gnolang/bench_preprocess_test.go`. It constructs a representative
Gno source file inline (exercising every reachable preprocess code path
that Gno allows â€” `chan`/`go`/`send`/`select` are rejected at parse time
and therefore excluded), and runs Preprocess on it once per iteration at
a unique pkg path. Per-code averages are emitted as benchmark metrics
(`ns/<CodeName>`).

The Makefile target invokes `cmd/calibrate/gen_preprocess_table.py`,
which medians the metrics across `-count=N` runs, applies the optional
`SCALE` factor, and writes:

- `cmd/calibrate/preprocess_gas_table.go.txt` â€” Go-pasteable
  `preprocessGasCosts` block.
- `cmd/calibrate/preprocess_gas_formulas.md` â€” per-code "flat" entry
  with measurement provenance (matches the `op_gas_formulas.md` /
  `native_gas_formulas.md` style).

This mirrors upstream's `gen_native_table.py` flow (added in #5291):
benchmark â†’ Python generator â†’ Go snippet + Markdown doc. The
preprocess benchmark is **not** parsed by `cmd/calibrate/gen_analysis.py`
â€” that script is specific to opcode handler benchmarks (`BenchmarkOp*`)
and fits parameterized cost shapes. Preprocess uses a separate flat
"per visit" model and therefore has its own dedicated generator.

This replaces the original PR's custom `cmd/benchops` binary (which
upstream removed in #5291) and aligns with the `bench_ops_test.go` /
`bench_gc_test.go` pattern used for other ops.

`consumePreprocessGas(gasMeter, code)` does the actual charge: looks up
the cost for `code` and calls `gasMeter.ConsumeGas(cost,
"Preprocess:<code>")`.

### Part D: Exhaustive code mapping with explicit invariants

`preprocessNodeCode(stage, n)` returns the code for a given (stage, Node
type) pair. The mapping is **exhaustive** for every node type that
Transcribe dispatches:

- **`TRANS_LEAVE`**: every type gets its own code. This is where the bulk
  of preprocess work happens, and cost variance per type is large
  (measured 122 ns for `*FuncDecl` vs 6079 ns for `*StructTypeExpr`).
- **`TRANS_ENTER`**: six types that do specific enter-stage work in
  `preprocess1` get dedicated codes; the rest (35 types) share
  `PreprocessEnter`. Justified empirically â€” these are no-op fall-through
  in `preprocess1`'s TRANS_ENTER switch and measure identically (~50â€“100
  ns). An invariant comment on the grouped case documents the contract:
  if any of these types gains per-type enter-stage work, it must be split
  out and given a dedicated code.
- **`TRANS_BLOCK`** / **`TRANS_BLOCK2`**: one code per BlockNode type
  that Transcribe visits at that stage.

### Part E: Debug-panic, prod-fallback

Calibration gaps must not crash the chain, but they must be caught early
during development. The design uses the existing compile-time `debug`
const (via the `debug` build tag):

- `preprocessNodeCode`: on an unknown stage or (stage, node) pair â€”
  **debug: panic** with an actionable message naming the missing case;
  **non-debug: fall back** to the stage-generic code
  (`PreprocessEnter` / `Block` / `Block2` / `Leave`).
- `consumePreprocessGas`: on a code with zero calibrated cost â€”
  **debug: panic** with an actionable message naming the uncalibrated
  code; **non-debug: charge `preprocessDefaultGasCost`**.

The fallback is **derived at package init from the table itself**:
`preprocessDefaultGasCost = max(preprocessGasCosts) Ă— 1.2`. This makes
the safety invariant self-maintaining â€” re-calibrating the table
automatically lifts the fallback if a code now exceeds it, so a
calibration gap can never become cheaper than the heaviest real visit
and therefore can never be exploited as an under-charge DoS vector.
A unit test (`TestPreprocessDefaultGasCostInvariant`) guards the
property against future hand-coded constants.

In the happy path (all codes calibrated), neither panic fires and neither
fallback is taken, so the behavior is identical in debug and non-debug
builds.

### Part F: Gas test fixture updates

Existing gas assertions in `gnovm/tests/files/gas/*.gno`,
`gno.land/pkg/sdk/vm/gas_test.go`, and five integration `.txtar` files
(`gc`, `gnokey_gasfee`, `restart_gas`, `simulate_gas`) were updated to
reflect the new per-tx gas consumption. Coin-balance assertions in
`gnokey_gasfee.txtar` follow directly from the new gas-fee deltas
(gas Ă— 1 ugnot / 1000 gas).

## Alternatives Considered

### A1. Keep the original `cmd/benchops` binary + build tags

The original PR added a `benchmarkingpreprocess` build tag and wired
preprocess into `cmd/benchops/main.go`. Upstream removed `cmd/benchops`
entirely in favor of `cmd/calibrate/` (Go-bench + Python analysis) and
100+ parameterized benchmarks in `bench_ops_test.go`.

Rejected: maintaining a parallel binary/build-tag infrastructure after
upstream deliberately retired it would accumulate tech debt. Aligning
with the new pattern (Go benchmark + `ReportMetric` per code) gives the
same capability with less surface area.

### A2. Mechanical merge-conflict resolution

Apply the 85 conflict resolutions onto the stale branch.

Rejected: `preprocess.go` alone had conflicts from a dozen upstream
commits, and the `benchops` API was refactored in a way that collided
with our typed-op extension. Reauthoring on current master produces
cleaner output (~10 files changed vs 85 conflicts) and picks up all
upstream fixes for free.

### A3. Zero fallback â€” always panic

A stricter stance: if a code has 0 cost or (stage, node) is unmapped,
panic regardless of build. Catches every calibration gap immediately but
turns any future calibration bug into a chain halt.

Rejected for chain safety: consensus nodes must not crash on a code bug
in a live network. Debug-only panic is sufficient to catch gaps during
development and CI; the prod fallback is insurance.

### A4. Split every node type at TRANS_ENTER

For consistency with TRANS_LEAVE, assign each of the 35 grouped
TRANS_ENTER types its own `PreprocessEnter*` code.

Rejected: the benchmark shows all 35 cost the same (~50â€“100 ns) because
`preprocess1` does no per-type work for any of them at enter. Splitting
would add 35 codes of identical cost â€” more surface area, no accuracy
win. An invariant comment on the grouped case was added instead, flagging
that if any of these types gains enter-stage work it must be split out.

## Consequences

### Positive

- **DoS closed.** Every preprocess visit is now gas-accounted. There is no
  code path that preprocesses for free.
- **Calibration from real measurements.** Costs are derived by running
  the actual preprocess pass on a representative corpus via Go
  benchmarks, not heuristics.
- **Loud calibration discipline.** Adding a new `PreprocessOp` without a
  cost entry is a debug-runtime panic. Adding a new node type to
  Transcribe without updating `preprocessNodeCode` panics in debug.
- **Chain-safe.** Production nodes never crash on a calibration gap; they
  charge a punitive default instead, so the worst-case behavior is a
  small gas over-charge on future unknown code, not a chain halt.
- **Aligned with upstream's bench pattern.** `BenchmarkPreprocess_Corpus`
  sits alongside `BenchmarkOpAdd_Int`, `BenchmarkOpAdd_String`, etc. in
  `pkg/gnolang/` â€” uses the same Go-benchmark + `ReportMetric`
  infrastructure that upstream standardized on in PR #5291.
- **No build-tag proliferation.** Timing is gated by a runtime bool
  (`preprocessTiming`), not a compile-time tag. Zero overhead in
  production (single branch-predicted check).

### Negative

- **Costs are workload-dependent.** Averages measured from one benchmark
  corpus won't match the exact profile of arbitrary user code. This is
  inherent to any cost table derived from sampling.
- **Hardware-dependent.** The checked-in values are from Apple M1 Pro;
  they should be regenerated on the canonical calibration machine
  (Xeon 8168, per `cmd/calibrate/`) before final release.
- **Instrumentation overhead in measurements.** `StartPreprocess` and
  `StopPreprocess` each call `time.Now()`. Measured cost on Apple
  M1 Pro: **~47 ns per call â†’ ~94 ns of timer overhead per visit**.
  This overhead is included in the calibrated costs but absent in
  production (instrumentation off = single branch-predicted check).
  Effect: small codes are over-charged by ~94 ns. For the smallest
  entries this is **larger than the underlying signal**:

  | Code | M1 ns measured | Of which is timer | "Real" work |
  |---|---:|---:|---:|
  | `PreprocessEnter` | ~60 | ~94 | ~0 (overhead-bound) |
  | `PreprocessBlockIfStmt` | ~98 | ~94 | ~4 |
  | `PreprocessLeaveStructTypeExpr` | ~6080 | ~94 | ~5990 |

  Sub-200 ns entries are essentially "two `time.Now()` calls" rather
  than real work. Direction is still **safe** (over-charge, never
  under-charge) â€” a contract dominated by `PreprocessEnter` events
  pays a small constant fee per visit, which is what we want for a
  per-visit DoS bound â€” but the table comment in `preprocess.go`
  should not be read as "this is exactly what the work costs."
- **Re-calibration is manual.** When `preprocess1` gains per-type work,
  the maintainer must notice and re-run the benchmark. The invariant
  comments document this but don't enforce it.
- **Gas assertions must follow.** Any future change to preprocess cost
  shifts per-tx gas; the gas-related test fixtures will need updates.
  Mitigated by localizing the fixtures to a small set of files.

### Neutral

- **Preprocess timer is independent of VM op timer.** `StartPreprocess`
  doesn't touch `curOpCode` â€” the two timelines run independently. This
  is intentional: preprocessing runs before VM execution, so there's
  never a time when both are active in the same goroutine.
