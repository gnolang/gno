# Native Function Gas Metering — Design Ideas

Today, every native function dispatched from GnoVM is charged a single flat
constant — `OpCPUCallNativeBody = 2205` (`gnovm/pkg/gnolang/machine.go:1266`,
flagged `// XXX arbitrary, not properly benchmarked`) — regardless of identity
or input size. `crypto/sha256.sum256(1MiB)`, `crypto/ed25519.verify`,
`time.now`, and `math.Float64bits` all cost the same. The stdlib Go impls
under `gnovm/stdlibs/**` never call `incrCPU` / `ConsumeGas` themselves.

The opcode side has a calibrated model (`gnovm/cmd/calibrate/`,
`op_gas_formulas.md`, 1 gas = 1 ns on Xeon 8168). Below are three distinct
designs for extending that discipline to the 45 entries in
`gnovm/stdlibs/generated.go`.

---

## Approach 1: Calibrated Per-Function Table

Mirror the existing opcode calibration model: every native binding gets one or
more entries in a code-generated gas table (analogous to OpCPU constants),
charged at the dispatcher (not inside the Go impl). Formula shape (flat /
`base + slope*N` / quadratic) is chosen per function, fit from real
benchmarks, regenerated by the same pipeline that already produces
`op_gas_formulas.md`.

### 1. Where charging happens

Today the only charge is the flat `OpCPUCallNativeBody = 2205` consumed in
`machine.go:1541` before `doOpCallNativeBody` runs. We replace that single
charge with a per-function lookup that runs *after* args are bound to the
block but *before* `nativeBody(m)` executes, so the dispatcher can read input
sizes off the block.

The cleanest seam is `doOpCallNativeBody` in `gnovm/pkg/gnolang/op_call.go:234`.
The frame's `*FuncValue` already carries enough identity (pkg + name) to index
the table, but to avoid a string lookup on every call, we extend `NativeFunc`
with a stable index:

```go
// gnovm/stdlibs/generated.go (regenerated)
type NativeFunc struct {
    gnoPkg     string
    gnoFunc    gno.Name
    nativeID   uint16        // NEW: index into nativeGasTable
    params     []gno.FieldTypeExpr
    ...
}
```

`*FuncValue.NativeID` is populated at bind time (same place `nativeBody` is
wired). Then:

```go
// op_call.go
func (m *Machine) doOpCallNativeBody() {
    fv := m.LastFrame().Func
    chargeNativeGas(m, fv) // NEW
    fv.nativeBody(m)
}

// gnovm/stdlibs/native_gas.go (generated)
func chargeNativeGas(m *gno.Machine, fv *gno.FuncValue) {
    e := nativeGasTable[fv.NativeID]
    cost := e.Base
    if e.SlopeIdx >= 0 {
        b := m.LastBlock()
        n := sizeOfParam(b, e.SlopeIdx, e.SlopeKind)
        cost += e.Slope * n
        if e.Quad != 0 {
            cost += e.Quad * n * n / e.QuadDiv
        }
    }
    m.IncrCPU(cost)
}
```

Knowing `N` is the load-bearing detail. The existing `params []FieldTypeExpr`
already tells us each parameter's gno type. For four representative cases:

| Native | SlopeIdx | SlopeKind | What we read |
|---|---|---|---|
| `sha256.sum256(data []byte)` | 0 | `lenBytes` | `b.Values[0].GetLength()` |
| `ed25519.verify(pub, msg, sig)` | 1 | `lenBytes` | `b.Values[1].GetLength()` (msg dominates) |
| `time.now()` | -1 | `flat` | nothing |
| `banker.sendCoins(..., coins Coins)` | 3 | `lenSlice` | `b.Values[3].GetLength()` |

`sizeOfParam` is a small switch over `SlopeKind` (`flat`, `lenBytes`,
`lenSlice`, `lenString`, `numCoins`, `bigIntBits`). Only ~6 kinds cover all 45
natives.

### 2. Populating the table

A new file `gnovm/cmd/calibrate/native_bench_test.go` benchmarks each native
by *calling the Go impl directly* (not the gno dispatcher — we want to isolate
the work, then add the fixed dispatcher overhead separately):

```go
func BenchmarkNative_SHA256Sum256(b *testing.B) {
    for _, n := range []int{32, 64, 256, 1024, 4096, 16384} {
        data := make([]byte, n)
        b.Run(fmt.Sprintf("n=%d", n), func(b *testing.B) {
            b.SetBytes(int64(n))
            for i := 0; i < b.N; i++ { sha256.Sum256(data) }
        })
    }
}

func BenchmarkNative_Ed25519Verify(b *testing.B) {
    pub, priv, _ := ed25519.GenerateKey(rand.Reader)
    for _, n := range []int{32, 256, 4096} {
        msg := make([]byte, n); sig := ed25519.Sign(priv, msg)
        b.Run(fmt.Sprintf("n=%d", n), func(b *testing.B) {
            for i := 0; i < b.N; i++ { ed25519.Verify(pub, msg, sig) }
        })
    }
}

func BenchmarkNative_TimeNow(b *testing.B)        { /* flat */ }
func BenchmarkNative_BankerSendCoins(b *testing.B){ /* vary len(coins) */ }
```

`gen_native_table.py` (sibling of `gen_analysis.py`) parses
`go test -bench=Native` output, fits one of three shapes per function (chosen
by lowest residual), and emits Go:

```python
# gen_native_table.py output -> native_gas.go
ENTRY("crypto/sha256",   "sum256",      slope_idx=0, kind="lenBytes",
      base=180, slope=12)         # fit: 178.4 + 11.7*N ns
ENTRY("crypto/ed25519",  "verify",      slope_idx=1, kind="lenBytes",
      base=58000, slope=2)        # ed25519 verify is ~constant; slope tiny
ENTRY("time",            "now",         kind="flat", base=42)
ENTRY("chain/banker",    "sendCoins",   slope_idx=3, kind="lenSlice",
      base=900, slope=210)
```

### 3. Assigning formula shapes

Hybrid, same as opcodes today: the fitter tries flat / linear / quadratic and
picks by adjusted R². But for natives we additionally accept a one-line
annotation in the genstd source comment (e.g.
`//gno:gas slopeParam=0 kind=lenBytes shape=linear`) that pins the choice.
This matters for the handful of functions whose cost depends on a non-obvious
arg (e.g. `banker.sendCoins` cost scales with `len(coins)`, not the address
string). Without the hint the fitter would have to try all params and we'd get
noise. Default (no hint, no benchmark) is `flat = OpCPUCallNativeBody` —
preserves current behavior for any new native added before its benchmark
lands.

### 4. The on-chain artifact

A second generated file `gnovm/stdlibs/native_gas.go`, indexed by the same
`nativeID` used in `generated.go`:

```go
// Code generated by gen_native_table.py; DO NOT EDIT.
package stdlibs

type nativeGasEntry struct {
    Base, Slope, Quad, QuadDiv int64
    SlopeIdx                   int8   // -1 = flat
    SlopeKind                  uint8  // kindFlat, kindLenBytes, ...
}

var nativeGasTable = [...]nativeGasEntry{
    /* 0  chain.packageAddress */          {Base: 310, SlopeIdx: -1},
    /* 1  chain.deriveStorageDepositAddr*/ {Base: 340, SlopeIdx: -1},
    /* 7  crypto/sha256.sum256 */          {Base: 180, Slope: 12,  SlopeIdx: 0, SlopeKind: kindLenBytes},
    /* 9  crypto/ed25519.verify */         {Base: 58000, Slope: 2, SlopeIdx: 1, SlopeKind: kindLenBytes},
    /* 18 time.now */                      {Base: 42,  SlopeIdx: -1},
    /* 31 chain/banker.sendCoins */        {Base: 900, Slope: 210, SlopeIdx: 3, SlopeKind: kindLenSlice},
    // ... 45 entries total
}
```

The order is locked to `nativeFuncs[...]` so the two arrays stay in sync (both
regenerated from the same genstd manifest).

### 5. Pros, cons, biggest risk

**Pros.** Reuses the calibration discipline already trusted for opcodes
(1 gas = 1 ns on Xeon 8168); charging is centralized so stdlib Go code stays
gas-unaware; cost of `sha256.Sum256(16KB)` finally reflects reality (~200x
today's flat 2205); easy to audit — `native_gas.go` is one screen of
constants.

**Cons.** Adds a coupling between `misc/genstd` and `cmd/calibrate` — genstd
must know about benchmark output, or at least about the annotation grammar.
`sizeOfParam` adds one indirection per native call (~5 ns), so very cheap
natives like `time.now()` get measurably more dispatcher overhead than work.
Three-shape menu is rigid: a function that's `O(N log N)` (e.g. sort) gets
approximated as linear with a fudged slope.

**Biggest risk.** *Calibration drift between Go versions.* `crypto/sha256` and
especially `crypto/ed25519` use Go's `asm` paths whose perf shifts noticeably
across Go releases (Go 1.21 vs 1.22 ed25519 got ~2x faster). Today's flat
constant is wrong but stable. A calibrated table is right today and silently
wrong after every toolchain bump unless the bench is part of CI and a regen
is part of the release checklist. Mitigation: gate `go test` releases on a
`make calibrate-check` that diffs the regenerated table against the committed
one and fails CI if any entry moves >20%, forcing a conscious bump (which is
itself a consensus-affecting change and must be versioned alongside hard-fork
height).

## Approach 2: In-Function Hand-Written Metering

Each `X_<name>` Go implementation in `gnovm/stdlibs/**` becomes responsible
for charging its own gas. It sees the *real* runtime inputs (slice lengths,
signature sizes, coin counts) and calls `m.GasMeter.ConsumeGas(...)` at
well-chosen points. The dispatcher's flat `OpCPUCallNativeBody = 2205`
shrinks to a tiny per-call dispatch overhead (we propose ~150 ns; see
`OpCPUPrecallFunc = 178` for comparison) and all *variable* cost moves into
the impl.

### 1. Plumbing `*gno.Machine` into every `X_<name>`

Today, `genstd` already supports a `MachineParam` flag
(`misc/genstd/mapping.go:312-326`): if the first Go param is `*gno.Machine`,
the wrapper passes `m` and the gno-side signature drops it. Of the 45
natives, ones like `X_now`, `X_emit`, and `X_bankerSendCoins` already take
`*gno.Machine`; pure-compute ones like `X_sum256`, `X_verify`,
`X_packageAddress` do not.

Cleanest path: **make `*gno.Machine` mandatory as the first arg of every
`X_*` function**. Reasons:

- The GasMeter lives on `*gno.Machine` (`m.GasMeter.ConsumeGas`); a separate
  `*store.GasMeter` parameter would duplicate plumbing and require touching
  every call site whenever the meter API evolves.
- `genstd`'s machine-detection logic already exists; we delete the conditional
  branch in `mapping.signaturesMatch` and require the param.
- The corresponding `.gno` declarations are `bodyless` and don't list `m`,
  so no Gno-side change is needed.

Representative signature changes (one trivial, one variable-cost, one
chain-state):

```go
// gnovm/stdlibs/time/time.go — was already hasMachine=true
func X_now(m *gno.Machine) (sec int64, nsec int32, mono int64) {
    m.GasMeter.ConsumeGas(gasTimeNow, "time.now")
    if m == nil || m.Context == nil { return 0, 0, 0 }
    ctx := execctx.GetContext(m)
    return ctx.Timestamp, int32(ctx.TimestampNano),
        ctx.Timestamp*int64(time.Second) + ctx.TimestampNano
}

// gnovm/stdlibs/crypto/sha256/sha256.go — gains *gno.Machine
func X_sum256(m *gno.Machine, data []byte) [32]byte {
    n := int64(len(data))
    // base + per-byte; SHA-256 is block-structured so use 64-byte blocks.
    blocks := (n + 63) / 64
    m.GasMeter.ConsumeGas(gasSHA256Base+blocks*gasSHA256PerBlock, "sha256.Sum256")
    return sha256.Sum256(data)
}

// gnovm/stdlibs/chain/banker/banker.go — was already hasMachine=true
func X_bankerSendCoins(m *gno.Machine, bt uint8, fromS, toS string,
    denoms []string, amounts []int64) {

    // Hashed map writes scale with #coins; charge before doing the work.
    nCoins := int64(len(denoms))
    m.GasMeter.ConsumeGas(
        gasBankerSendBase + nCoins*gasBankerSendPerCoin,
        "banker.SendCoins")

    ctx := execctx.GetContext(m)
    amt := CompactCoins(denoms, amounts)
    /* ... unchanged ... */
}
```

For `X_verify` (ed25519): charge once before calling, since `ed25519.Verify`
is essentially constant-time per-public-key with cost dominated by message
length:

```go
func X_verify(m *gno.Machine, publicKey, message, signature []byte) bool {
    m.GasMeter.ConsumeGas(gasEd25519Base+int64(len(message))*gasEd25519PerByte,
        "ed25519.Verify")
    return ed25519.Verify(publicKey, message, signature)
}
```

### 2. Where the cost formula lives

A sibling `gas.go` per stdlib package, keeping constants out of the impl body
but next to it for review:

```go
// gnovm/stdlibs/crypto/sha256/gas.go
package sha256

// Derived from cmd/calibrate/sha256_bench_test.go on Xeon 8168.
// y = 73ns + 240ns/block (R^2 = 0.998).
const (
    gasSHA256Base     = 73
    gasSHA256PerBlock = 240
)
```

```go
// gnovm/stdlibs/chain/banker/gas.go
const (
    gasBankerSendBase    = 9_500   // KVStore Get+Set+marshal overhead
    gasBankerSendPerCoin = 1_200   // per coin map write
)
```

This places the cost expression next to the impl (locality) but in a separate
file so a stdlib refactor PR's diff doesn't accidentally hide a pricing change
— reviewers see `gas.go` touched.

### 3. How constants are picked / tuned

- **Pure CPU natives** (`sha256`, `ed25519`, `bech32`): derived from
  `cmd/calibrate` micro-benchmarks (model: `1 gas = 1 ns on Xeon 8168`, see
  `op_bench_analysis.txt`). Add a `bench_test.go` per stdlib pkg following
  the pattern in `gnovm/cmd/calibrate/sha256_bench_test.go`. A new analysis
  stage in `gen_analysis.py` emits a `gas_natives.csv`; the stdlib author
  copies fitted base+slope into `gas.go`.
- **Chain-state natives** (`banker.*`, `chain.emit`, `params.*`): cannot be
  benchmarked in isolation because cost depends on the underlying KVStore.
  Hand-tuned with conservative upper-bound estimates, validated against the
  existing `OpCPUCallNativeBody = 2205` baseline so we don't regress.
- **Ownership**: the stdlib author writes `gas.go`; a `CODEOWNERS` rule
  routes any `gas.go` change to the gas-economics reviewers (similar to how
  `machine.go`'s `OpCPU*` constants are gatekept today).

### 4. genstd impact

Minimal. The wrapper template at `misc/genstd/template.tmpl:78-85` already
passes `m` when `MachineParam` is true. Once we make `*gno.Machine`
mandatory:

- Delete the `if MachineParam` branch in `mapping.signaturesMatch` / drop the
  conditional from the template.
- The wrapper becomes uniform: `pkg.X_fn(m, p0, p1, ...)`.
- `incrCPU(OpCPUCallNativeBody)` in `op_call.go:doOpCallNativeBody` is
  replaced with `incrCPU(OpCPUCallNativeDispatch)` (~150).
- No template logic for gas — it's purely a side effect inside the Go fn.

### 5. Edge case: panic mid-call

Charge **before** doing the work. With the pattern above, `X_verify` consumes
gas first, then runs the verifier; if `ed25519.Verify` returns false, gas was
already paid (correct — work was done). If `X_emit` panics on odd-attr count,
gas was charged for the dispatch but not the emit work — acceptable since the
panic propagates as a Gno panic and the tx aborts. The dangerous failure mode
is: charge *after* the work, then a panic short-circuits payment for an
expensive op. Convention: **always charge the bound first, then do the work**.
For natives where the formula needs the Go fn to inspect input first (e.g.
variable-length parsing), do an O(1) input-size check, charge, then call.

### 6. Pros / cons / risks

- **Pros**: cost expression sits next to the impl, so a stdlib author who
  changes the algorithm sees the gas constants in the same package; access to
  actual runtime sizes (slice lengths, map cardinalities) without reflection
  in the dispatcher; supports complex nonlinear costs (e.g. `n log n` for
  sort) trivially.
- **Cons**: 45 hand-written formulas to audit and keep in sync with
  calibration; each stdlib PR can silently change pricing if review is lax;
  no central registry to grep — gas math is scattered across
  `gnovm/stdlibs/**/gas.go`.
- **Biggest risk**: **forgotten metering** in a new native. With
  dispatcher-side metering you cannot ship an unmetered native; with this
  approach a new `X_foo` that omits `m.GasMeter.ConsumeGas` compiles, links,
  and runs free until someone notices. Mitigation: a `go vet`-style linter in
  `misc/genstd` that fails the build if any `X_*` function body lacks a call
  to `m.GasMeter.ConsumeGas` (or is explicitly annotated `// gnoGas:flat` to
  opt into a default 150 ns charge).

## Approach 3: Declarative Gas Metadata in genstd

The codegen pipeline at `misc/genstd` already walks every native binding
(`mapping.go:46 linkFunctions`) and emits a wrapper `f` per function in
`gnovm/stdlibs/generated.go` (template at `misc/genstd/template.tmpl:50-99`).
That wrapper is the *only* place every native call funnels through. We can
teach genstd to read a per-native gas spec and emit the gas charge inline,
before `X_<name>` is invoked, using the parameter symbols (`p0`, `p1`, ...)
that the template already binds. The runtime keeps a single helper
(`m.incrCPU`); the Go stdlib impls (`gnovm/stdlibs/crypto/sha256/sha256.go`,
etc.) stay untouched.

### 1. Directive syntax

A `//gno:gas` line directly above the Go `X_` declaration. The shape mirrors
the calibration formulas in `cmd/calibrate/op_gas_formulas.md` (`flat`,
`base + slope*N`, `base + slope*N1*N2`). Sizes are expressed as functions of
the named params `p0..pN`, with a small allowlist: `len(pi)`, `bits(pi)`
(for `bigint`), `sum_len(pi)` (sum of lengths over a slice of strings).

```go
// gnovm/stdlibs/crypto/sha256/sha256.go
//gno:gas base=120 slope=12 size=len(p0)
func X_sum256(data []byte) [32]byte { return sha256.Sum256(data) }

// gnovm/stdlibs/crypto/ed25519/ed25519.go
//gno:gas flat=185000        // ed25519 verify is ~185us; signature length is fixed at 64
func X_verify(publicKey, message, signature []byte) bool { ... }

// gnovm/stdlibs/time/time.go
//gno:gas flat=40
func X_now(m *gno.Machine) (sec int64, nsec int32, mono int64) { ... }

// gnovm/stdlibs/chain/emit_event.go
//gno:gas base=900 slope=8 size=sum_len(p1)+len(p0)
func X_emit(m *gno.Machine, typ string, attrs []string) { ... }

// gnovm/stdlibs/chain/banker/banker.go
//gno:gas base=1500 slope=320 size=len(p3)   // p3 = denoms; 1 entry == 1 SendCoins coin
func X_bankerSendCoins(m *gno.Machine, bt uint8, fromS, toS string, denoms []string, amounts []int64) { ... }
```

Sizes are evaluated against the *Gno-side* arguments, which is exactly what
`genstd`'s template already has via `b.GetPointerTo(...).TV` for typed-value
params and via `pN` Go-typed locals for converted ones. For sized types we
always go through the TV (cheap: `tv.GetLength()` for slices/strings,
`tv.GetBigInt().BitLen()` for bigint), so the size charge runs *before* the
Gno→Go reflect conversion — important so a 1 GB string can't grief us through
reflection cost we never charged for.

### 2. Generated wrapper code

The template gains one block (`{{ if $m.Gas }} ... {{ end }}`) just below the
`b := m.LastBlock()` line. For sha256:

```go
// generated.go (sha256.X_sum256)
func(m *gno.Machine) {
    b := m.LastBlock()

    // --- emitted from //gno:gas base=120 slope=12 size=len(p0) ---
    {
        tv0 := b.GetPointerTo(nil, gno.NewValuePathBlock(1, 0, "")).TV
        n := int64(tv0.GetLength()) // len(p0)
        m.incrCPU(120 + 12*n)
    }
    // --- end gas ---

    var (
        p0  []byte
        rp0 = reflect.ValueOf(&p0).Elem()
    )
    tv0 := b.GetPointerTo(nil, gno.NewValuePathBlock(1, 0, "")).TV
    tv0.DeepFill(m.Store)
    gno.Gno2GnoValue(tv0, rp0)

    r0 := sha256.X_sum256(p0)
    m.PushValue(gno.Go2GnoValue(m.Alloc, m.Store, reflect.ValueOf(&r0).Elem()))
},
```

And for `chain.X_emit` (machine-param + sum-of-string-lengths):

```go
// generated.go (chain.X_emit)
func(m *gno.Machine) {
    b := m.LastBlock()

    // --- emitted from //gno:gas base=900 slope=8 size=sum_len(p1)+len(p0) ---
    {
        tv0 := b.GetPointerTo(nil, gno.NewValuePathBlock(1, 0, "")).TV // typ string
        tv1 := b.GetPointerTo(nil, gno.NewValuePathBlock(1, 1, "")).TV // attrs []string
        n := int64(tv0.GetLength())
        n += gno.SumStringLengths(tv1) // small helper in gnolang, O(len(slice))
        m.incrCPU(900 + 8*n)
    }
    // --- end gas ---

    // ... existing reflect conversion ...
    chain.X_emit(m, p0, p1)
},
```

The helper `SumStringLengths` is the only new runtime dependency — a 5-line
function next to `incrCPU`. Everything else is just `incrCPU` with arithmetic.

### 3. From calibration to coefficients

Calibration lives where it already lives: `gnovm/cmd/calibrate/`. We add a
sibling to `sha256_bench_test.go` — `native_bench_test.go` — that runs each
native at several sizes and fits the same shapes the analyzer already emits
for opcodes. Output is a checked-in `gnovm/cmd/calibrate/native_gas.csv` with
columns `pkg, func, shape, base, slope, size_expr, ref_hw, source_commit`.

The directive declares **shape and size expression**; the **coefficients are
looked up from the CSV by `(pkg, func)`** at codegen time. This separation
matters because:

- Recalibrating on new reference hardware should be a one-line table edit and
  a `go generate`, not a sweep through 45 source files.
- The shape is a semantic property of the function (sha256 *is* linear in
  input size); the slope is a hardware property.
- Reviewers diffing the CSV see only numbers; reviewers diffing source see
  only intent.

For the rare case where a value is *not* benchmarkable (e.g. `time.now`,
where 40 is just measured wrapper overhead) the directive can carry the
literal: `//gno:gas flat=40 source=manual`. genstd refuses to mix: either
both `base=` and `slope=` are literal, or both come from the table.

### 4. Where the directive lives — argued

**On the `X_` function declaration as a `//gno:gas` comment.** Not a sidecar
`gas.yaml` per package, not a central `misc/genstd/native_gas.toml`.

Reasons, in order of weight:

1. **Locality.** When you change `X_emit`'s signature (add a param, change a
   `string` to `[]byte`) you must update the size expression in the same
   diff. Sidecars rot; comments don't get past code review unchanged.
2. **One source of truth for shape.** The Go file already declares param
   names and types; the directive references those exact names (`p0`, `p1`).
   A central TOML would have to re-declare them.
3. **genstd already parses these files via `go/ast`.** Picking up `Doc`
   comments costs nothing; parsing a separate file means more I/O, more error
   reporting paths, and the package-discovery code in `walk.go` doesn't
   currently look at non-Go files.
4. **CSV stays central anyway** for the coefficients (point 3), so the
   "central registry" argument is satisfied for the part that actually
   benefits from being central.

The one case where a sidecar wins is bulk recalibration sweeps — but those
happen in the CSV, not the directive.

### 5. Unannotated natives

**Build error.** genstd panics with `package "chain": native function X_foo
has no //gno:gas directive`. This forces every new native to come with a
thought-through cost; the current `OpCPUCallNativeBody = 2205 // XXX
arbitrary` exists precisely because there was no forcing function. To unblock
prototyping there's an explicit escape hatch — `//gno:gas todo` — which emits
`m.incrCPU(OpCPUCallNativeBody)` and is greppable for cleanup. CI runs
`grep -r "gno:gas todo" gnovm/stdlibs` and fails on release branches.

### 6. Pros, cons, biggest risk

**Pros**

- Zero runtime overhead beyond the `incrCPU` itself (no reflection-driven
  dispatch, no map lookup per call, no per-call closure construction).
- Fully auditable: the diff to `generated.go` shows exactly which charge each
  function takes, in plain Go.
- Calibration coefficients and Go impls evolve independently: rerun
  benchmarks → edit CSV → regenerate. No impl change needed.
- Size extraction happens *before* Gno→Go conversion, so adversarial inputs
  are charged before they cost real CPU.
- Fits the project's existing aesthetic — `op_gas_formulas.md` already uses
  this exact taxonomy.

**Cons**

- genstd grows: directive parser, size-expression mini-language, CSV joiner,
  helper-emitter. Roughly +400 LOC.
- Directive schema needs versioning. Recommend a header `//gno:gasv 1` per
  file the first time the directive appears, rejected mismatches at codegen
  time.
- The size mini-language is a new surface area — easy to want `len(p0[0])`
  next, then we're building a real expression evaluator. Hard cap: only the
  four primitives listed (`len`, `bits`, `sum_len`, integer arithmetic with
  `+ * constants`). Anything more exotic falls back to `flat` plus a TODO.

**Biggest risk: silent miscalibration after a signature change.** If someone
reorders params in `X_bankerSendCoins`, `len(p3)` now measures the wrong
slice and gas is wrong but compiles fine. Mitigation: the directive can name
the param (`size=len(denoms)`) and genstd cross-checks against the Go AST.
Codegen fails if `denoms` isn't a param of the function. This is cheap to
implement and closes the only failure mode that's both plausible and
undetectable by unit tests.

# Prior Art — gnolang/gno PRs and Issues

Survey of what the upstream project has already shipped, attempted, or
explicitly deferred around this problem. The pattern across these is
consistent: opcode-level and store-level metering exist; **per-native cost
parameterization is openly acknowledged as missing and has been deferred at
least twice**.

## The smoking gun: PR #2149 (GasUsed for std)

[PR #2149 — `feat(Gnovm/std): GasUsed function for std`](https://github.com/gnolang/gno/pull/2149)
(closed in favor of #2571) initially had the new native consume a
`defaultInvokeCost` on call. Reviewer @thehowl removed it with the rationale:

> *"Don't consume gas here; we'll need to benchmark all native functions and
> add a cost on another occasion."*

This is the explicit deferral that produced the current state — every native
billing the same flat `OpCPUCallNativeBody = 2205 // XXX arbitrary`. None of
the follow-ups (#2571, #4953, #5087) closed that loop.

## PR #1430 — consolidate VM gas consumption (piux2)

[PR #1430](https://github.com/gnolang/gno/pull/1430) is the foundation of
today's metering: it introduced the integrated gas meter that combines CPU
cycles, allocation, store access, tx size, and signature verification, plus
the `GasFactorCPU` linkage between cycles and gas. It establishes
`incrCPU()` per opcode — the surface our three approaches all extend — but
treats native calls as opaque `OpCallNativeBody` opcodes. A review comment
on the PR also floated a micro-opt: *"look up the CPU cost per opcode and
call `incrCPU()` once before the switch"* — relevant to Approach 1, which
adds a second indirection per native call and would benefit from the same
hoist.

## PR #4953 — centralize gas costs (closed Apr 2026)

[PR #4953](https://github.com/gnolang/gno/pull/4953) (referenced as "ready"
by issue #5087) moves all gas constants into `tm2/pkg/gas` (`operation.go`
+ `config.go`). Crucially it introduces the **first per-unit native charge
in the codebase**:

> *"If a cost is per unit (like printing bytes), the usage is
> `gasMeter.GasConsume(OpNativePrintPerByte, len(output))`."*

This is a one-off precedent for Approach 2's pattern (charge inside the
native impl with `len(input)`), applied only to native print. It does not
generalize to the other 44 natives. Reviewers also flagged that the PR uses
`float64` for cost multipliers — a consensus-determinism red flag (Gno uses
softfloat for the same reason). Any of our three approaches needs to keep
arithmetic in `int64`.

## Issue #5087 — META: Gas improvements (open, milestone Beta Bug Bash)

[Referenced from #5089 Beta Launch Checklist](https://github.com/gnolang/gno/issues/5089).
Owner: @aeddi. Lists four work items, two of which are direct hooks for
this work:

- ✅ Centralize gas costs (PR #4953, ready)
- ⏳ Documentation clarification (#5086)
- ⏳ **Benchmarks** (TODO)
- ⏳ **Missing gas charges** (TODO)

Native function calibration falls under both unfilled buckets. None of our
three approaches conflicts with #4953's centralization — Approach 1's
`nativeGasTable` and Approach 3's emitted `incrCPU` calls would live
naturally next to the new `tm2/pkg/gas` constants.

## PR #859 — improved native bindings (thehowl, merged)

[PR #859](https://github.com/gnolang/gno/pull/859) is the architectural
ancestor of `genstd`. It replaced runtime `DefineNative` injection with
static codegen, introducing `NativePkg`/`NativeName` metadata on
`*FuncValue`, the `NativeStore` resolver, and the `X_*` Go-side
implementation convention. **No gas changes.** This PR is the reason
Approach 3 is even feasible — the codegen seam already exists; we'd just
extend the template.

## PR #2571 — gas consumption in `gno test` output (closed)

[PR #2571](https://github.com/gnolang/gno/pull/2571) and its predecessor
[PR #783](https://github.com/gnolang/gno/pull/783) (`gno test
-print-runtime-metrics`, by moul) wire `m.GasMeter.GasConsumed()` into test
output. Both are *consumers* of metering, not producers. Once per-native
charges land, these immediately surface the cost to realm authors — a good
forcing function for calibration realism.

## Related: out-of-gas surface (Issue #2365)

[Issue #2365](https://github.com/gnolang/gno/issues/2365) — "out of gas
error is unclear `(VM call panic {WritePerByte})`" — confirms that
per-byte charging *is* already used inside the store layer, and that the
`ConsumeGas(amount, "name")` descriptor surfaces in panics. Both Approach 2
(in-function metering) and Approach 3 (codegen-emitted) should pass
descriptive names like `"sha256.Sum256"` so the same mechanism gives realm
authors useful diagnostics.

## Synthesis

The upstream picture, in one paragraph: opcode metering is calibrated and
trusted (`OpCPU*` constants, `cmd/calibrate`); store metering is per-byte
and battle-tested (`WritePerByte`, etc.); a single native (`print`) got
per-byte charging in #4953; everything else under `gnovm/stdlibs/**`
shares one flat XXX-marked constant. The maintainers know — see the
@thehowl quote on #2149 and the "Missing gas charges" TODO on #5087 —
they just haven't done it. Of the three approaches above, **Approach 3
(declarative metadata in genstd)** is the closest match to existing
project aesthetic: it reuses `genstd` (#859), inherits the formula taxonomy
from `op_gas_formulas.md`, slots into the `tm2/pkg/gas` constant home
defined by #4953, and avoids the "stdlib PR can change pricing" risk that
worried reviewers on #2149.
